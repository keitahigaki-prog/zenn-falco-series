# 第11章 運用設計と組織展開

## はじめに

ここまでの章では、個々のルールチューニングテクニックを学んできました。しかし、本当の課題はこれからです。Falcoは組織全体で運用される検知基盤です。「誰が、どのタイミングで、どのプロセスを通じてルールを変更するのか」という運用設計がなければ、せっかくチューニングしたルールも台無しになってしまいます。

本章では、SRE・Platform・Security・Development チームが協働してFalcoを運用していく実践的な組織モデルを解説します。

---

## 1. ルール変更の権限と責任（RACI）

まず整理すべきは「誰が何を決めるのか」という責任分担です。RACI（Responsible, Accountable, Consulted, Informed）マトリクスを使うと以下のようになります。

```
┌─────────────────────────┬─────────┬──────────┬──────────┬──────┐
│ アクティビティ           │ SRE     │ Platform │ Security │ Dev  │
├─────────────────────────┼─────────┼──────────┼──────────┼──────┤
│ ルール新規作成           │ R/A     │ C        │ C        │ I    │
│ ノイズ削減の例外追加     │ R       │ A        │ C        │ C    │
│ セキュリティルール修正   │ C       │ R        │ A        │ I    │
│ 本番への段階的導入       │ R/A     │ C        │ C        │ -    │
│ 例外の棚卸し・削除       │ R       │ A        │ C        │ -    │
│ アップストリーム更新     │ R/A     │ C        │ I        │ -    │
│ dev環境での実験          │ R       │ C        │ C        │ A    │
│ ルール設定のドキュメント │ C       │ R/A      │ C        │ C    │
└─────────────────────────┴─────────┴──────────┴──────────┴──────┘
```

本章では、変更承認フロー、GitOpsによるルール管理、例外の期限管理、定期棚卸し、本番反映フロー、KPIの定義などを詳しく解説しています。

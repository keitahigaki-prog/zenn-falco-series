# 第7章 誤検知・過検知・見逃しの調査フロー

## はじめに

Falcoのアラートを受け取った直後、エンジニアが最初にやるべきことは何でしょうか。ただ「リスクがある」と判断して対応するのではなく、**そのアラートが正しい検知か、それとも誤検知か**を冷静に判断することです。

本章では、アラートを受け取ったときの初動フロー（型）を解説します。これを身につけることで、チーム内で一貫した調査ができるようになり、同じ誤検知で何度も対応する無駄を減らせます。

## 7.1 初動フロー：4つのStep

アラートを受け取ったときの調査は、以下の4Stepで進めます。

```
Step 1: outputを読む
    ↓
Step 2: 正常か異常かの判断
    ↓
Step 3: conditionのどの部分にマッチしたかを特定
    ↓
Step 4: どの粒度で修正するかを意思決定
```

### Step 1: outputを読む — フィールド値の確認

まずはアラートメッセージの**すべてのフィールド値**を読み込みます。これは「見る」ではなく「読む」ことが大切です。

**例：実際のアラート出力**

```
2026-04-05T14:32:10.123456Z: Warning Unauthorized process started
  (user=root command=/usr/bin/apt update container_id=a1b2c3d4)
```

ここから以下を抽出します：

| フィールド | 値 | 検証 |
|-----------|---|-----|
| user | root | ルートユーザーか |
| command | /usr/bin/apt update | バージョン管理外か |
| container_id | a1b2c3d4 | 本番環境か開発環境か |
| timestamp | 14:32:10 | メンテナンス時間か |

本章では、誤検知・過検知・見逃しの具体的な例を含め、調査フローと対処方法を詳しく解説しています。

ここまでの章で、OpenCTIを使って「脅威を知る」方法を学びました。しかし **「知っている」と「防げる」は別の話** です。

「うちのネットワークにはSalt Typhoonの攻撃手法を検知するルールがあるのか？」「EDRはMITRE ATT&CKのT1059（コマンドラインインターフェース）を本当にブロックできるのか？」 — こうした問いに答えるのが **OpenAEV（Open Adversarial Exposure Validation）** です。

## 5.1 OpenAEVとは

OpenAEVは、Filigranが開発する **攻撃シミュレーション・防御検証プラットフォーム** です。もともと **OpenBAS（Open Breach and Attack Simulation）** という名称で2024年初頭にOSSとしてリリースされ、2026年1月に **OpenAEV** にリブランドされました。

一言でいうと、OpenCTIが「脅威インテリジェンスを集めて理解する」プラットフォームなのに対し、OpenAEVは「その脅威インテリジェンスに基づいて、自社の防御を実際にテストする」プラットフォームです。

```
OpenCTI（知る）          OpenAEV（試す）
┌──────────────┐       ┌──────────────────┐
│ 脅威アクター   │──────→│ 攻撃シナリオ       │
│ マルウェア     │──────→│ インジェクト（攻撃手順）│
│ 脆弱性(CVE)   │──────→│ Nuclei CVEスキャン  │
│ TTP(ATT&CK)  │──────→│ エージェント実行     │
│ IOC          │       │                  │
└──────────────┘       └────────┬─────────┘
                               │
                        検証結果をフィードバック
                               │
                               ▼
                    ┌──────────────────┐
                    │ 予防 ✅ / 検出 ✅   │
                    │ 予防 ❌ / 検出 ❌   │
                    │ → ギャップを特定    │
                    └──────────────────┘
```

## 5.2 BAS（Breach and Attack Simulation）とは

OpenAEVが属するカテゴリは **BAS（Breach and Attack Simulation）** と呼ばれます。従来のペネトレーションテストとの違いを理解しておきましょう。

| | ペネトレーションテスト | BAS（OpenAEV） |
|---|-----|-----|
| **実行者** | 専門のペンテスター（外部委託が多い） | セキュリティチームが自分で実行 |
| **頻度** | 年1-2回 | いつでも（日次〜週次も可能） |
| **範囲** | テスターの技量に依存 | 定義済みシナリオで網羅的 |
| **コスト** | 高額（1回数百万円〜） | OSS版は無料 |
| **再現性** | テスターごとに異なる | 同じシナリオを何度でも再現 |
| **自動化** | 難しい | APIで完全自動化可能 |

BASの最大の利点は **継続的に防御を検証できる** ことです。インフラ変更やルール更新のたびにテストを回し、退行（リグレッション）がないか確認できます。

## 5.3 Threat-Informed Defenseの概念

セキュリティ業界では、「脅威を知る → 防御を検証する → ギャップを埋める」のサイクルを **Threat-Informed Defense（脅威情報に基づく防御）** と呼びます。

```
    ┌─────────────┐
    │  1. 脅威を知る │ ← OpenCTI
    │  (Understand) │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 2. 防御を検証  │ ← OpenAEV
    │  (Validate)   │
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 3. ギャップを  │ ← SIEM/EDR/Firewall
    │  埋める (Fix)  │     ルール修正
    └──────┬──────┘
           │
           ▼
    ┌─────────────┐
    │ 4. 再検証する  │ ← OpenAEV（再テスト）
    │ (Re-validate) │
    └──────┬──────┘
           │
           └──→ 1. に戻る（継続サイクル）
```

従来は「知る」と「検証する」が完全に分断されていました。この分断を、OpenCTIとOpenAEVの統合で一気通貫にするのがFiligranの **XTM（eXtended Threat Management）** 構想です。

:::

## 5.5 OpenCTIとOpenAEVの比較

| | OpenCTI | OpenAEV |
|---|---------|---------|
| **目的** | 脅威を知る・理解する | 防御を検証する |
| **主な対象** | 脅威アクター、マルウェア、CVE、IOC | 攻撃シナリオ、防御ルール、エンドポイント |
| **データモデル** | STIX 2.1 | シナリオ / インジェクト / エージェント |
| **出力** | 脅威インテリジェンスレポート | 防御ギャップレポート、スコアカード |
| **利用者** | CTIアナリスト、SOCアナリスト | レッドチーム、SOCアナリスト、セキュリティエンジニア |
| **技術スタック** | GraphQL / Elasticsearch / Redis | REST API / PostgreSQL |
| **デモ環境** | demo.opencti.io | demo.openaev.io |

## 5.7 シナリオとは

OpenAEVの中核概念は **シナリオ（Scenario）** と **インジェクト（Inject）** です。

シナリオは「どんな攻撃を再現するか」を定義したテンプレートです。1つのシナリオは「特定の脅威アクターの攻撃フロー」や「特定のセキュリティ課題のテスト」を表現します。

### シナリオの種類

デモ環境には16個のシナリオが用意されています。大きく以下のカテゴリに分類できます。

**攻撃シナリオ（Attack Scenario）** — 実在のAPTグループや攻撃手法を再現する。

| シナリオ名 | 深刻度 | 内容 |
|----------|-------|------|
| [SALT TYPHOON] CVE & TTPs validation | HIGH | 中国系APTの攻撃手法再現 |
| 0ktapus | HIGH | フィッシングキャンペーン再現 |
| [Windows] All Payloads | MEDIUM | Windows向け全ペイロードテスト |
| [Linux] All Payloads | MEDIUM | Linux向け全ペイロードテスト |
| EASM Scenario | HIGH | 外部攻撃面管理テスト |

**グローバル危機（Global Crisis）** — 組織全体を巻き込む大規模インシデントのシミュレーション。

| シナリオ名 | 深刻度 | 内容 |
|----------|-------|------|
| Tabletop Exercise - Akira Ransomware Crisis | HIGH | ランサムウェア攻撃の卓上演習 |
| Dino Crisis | HIGH | 大規模セキュリティ危機の演習 |

:::message
**攻撃シナリオ vs グローバル危機の違い**
攻撃シナリオは技術的なテスト（実際にコマンドを実行）が中心です。グローバル危機は、技術だけでなく組織の対応プロセス（経営判断、広報対応、法務対応など）も含めた総合演習です。
:::

## 5.8 インジェクトとは

インジェクトはシナリオ内の **個別の攻撃ステップ** です。映画のシナリオに例えれば、シナリオが「脚本全体」で、インジェクトが「1つ1つのシーン」です。

### インジェクトの3つのタイプ

**1. 技術的インジェクト（Technical Inject）** — エージェントを通じて、テスト対象のエンドポイントで **実際にコマンドを実行** します。ファイルのダウンロード・実行、権限昇格、ラテラルムーブメント、レジストリ操作などが含まれます。

**2. 脆弱性スキャン（Nuclei CVE Scan）** — **Nuclei** テンプレートを使って、対象サーバーに特定のCVEが存在するかをスキャンします。ネットワーク経由でスキャン（エージェント不要）。

:::message
**Nucleiとは**
ProjectDiscoveryが開発するOSSの脆弱性スキャナーです。YAML形式のテンプレートで脆弱性チェックを定義でき、OpenAEVはこのテンプレートを使ってCVEの存在確認を自動化しています。
:::

**3. 人的インジェクト（Human Inject）** — 技術ではなく **人間の行動** をテストします。フィッシングメールの送信テスト、ソーシャルエンジニアリング、インシデント対応手順の遵守確認など。

## 5.9 Salt Typhoonシナリオの深堀り

Salt Typhoonは2019年から活動が確認されている中国系APTグループで、米国、東南アジア、アフリカの通信・政府機関を主な標的としています。

### 18個のインジェクト構成

このシナリオは、Salt Typhoonの攻撃を2つのフェーズで再現する18ステップで構成されています。

```
[SALT TYPHOON] CVE & TTPs validation — 18 インジェクト
│
├── Phase 1: 初期侵入の検証（Nuclei CVEスキャン）
│   ├── CVE-2024-20353 — Cisco ASA/FTDの脆弱性
│   ├── CVE-2021-26855 — Microsoft Exchange (ProxyLogon)
│   ├── CVE-2023-46805 — Ivanti Connect Secure
│   ├── CVE-2023-35078 — Ivanti EPMM
│   ├── CVE-2023-20198 — Cisco IOS XE
│   └── CVE-2023-48788 — Fortinet FortiClient EMS
│
└── Phase 2: ポストエクスプロイト（エージェント実行）
    ├── Download beacon to target
    ├── Deploy cab file containing beacon
    ├── Copy beacon cab file to remote target
    ├── Expand cab file to remote target
    └── ... 他のTTP再現ステップ（計12ステップ）
```

:::message alert
Salt Typhoonが悪用するCVEはいずれもネットワーク境界のデバイス（VPN、ファイアウォール、メールサーバー）を対象としています。これらは「侵入の入り口」となるため、パッチ適用の優先度は最高です。
:::

### Phase 2 で見えること

Phase 2は「侵入された後、内部でEDR/NDRが検知・ブロックできるか」を検証します。

| ATT&CK ID | テクニック名 | OpenAEVでの再現 |
|-----------|-----------|--------------|
| T1059 | Command and Scripting Interpreter | PowerShellコマンド実行 |
| T1547 | Boot or Logon Autostart Execution | 永続化メカニズムの作成 |
| T1548 | Abuse Elevation Control Mechanism | 権限昇格の試行 |
| T1562 | Impair Defenses | セキュリティツールの無効化試行 |

各ステップでEDRが「ブロック」「アラート」「スルー」のどれを返したかが、そのまま **防御の成績表** になります。

## 5.10 検証結果の読み方

シナリオ実行後、各インジェクトに対して以下のステータスが付与されます。

| ステータス | 意味 | 深刻度 |
|----------|------|-------|
| **たぶん防げた** | セキュリティツールがブロックした可能性が高い | 低（防御OK） |
| **インジェクト実行** | インジェクトが実行された（検知/ブロックされず） | **高（要対策）** |
| **不足内容** | 前提条件が不足（エージェント未設定など） | - |

「インジェクト実行」が出た場合は **防御にギャップがある** ことを意味します。

:::message
**結果は4軸で評価される**
OpenAEVは各インジェクトを以下の4軸で評価します。
1. **予防（Prevention）** — 攻撃を事前に防げたか
2. **検出（Detection）** — 攻撃を検知できたか
3. **脆弱性（Vulnerability）** — 脆弱性が存在するか
4. **人間の対応（Human Response）** — 人的な対応は適切だったか
:::

## 5.11 シナリオの作成とインポート

### XTMハブからのインポート

最も簡単な方法は **Filigran XTMハブ** からのインポートです。シナリオ一覧画面の右上にある「ハブからのインポート」ボタンをクリックすると、最新のシナリオをワンクリックで取り込めます。

### 手動でのシナリオ作成

シナリオ一覧画面右下の「＋」ボタンから新規作成も可能です。

1. シナリオ名、説明、深刻度、カテゴリーを設定
2. キルチェーンフェイズを選択
3. インジェクトを1つずつ追加
4. インジェクトの実行順序（依存関係）を設定
5. 対象のエージェント/ターゲットを指定

OpenCTIで調査した脅威アクターのTTPを基に、自社固有のシナリオを作成することで、**自社に最も関連性の高い攻撃**に対する防御を重点的にテストできます。

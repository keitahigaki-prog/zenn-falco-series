前章でシナリオとインジェクトの構造を学びました。この章では、実際にテストを実行するための **エージェント** と、個別技法をピンポイントでテストする **アトミックテスト**、そして **OpenCTIとOpenAEVの統合ワークフロー** を解説します。

## 6.1 エージェントの役割

エージェントは、テスト対象のエンドポイント（サーバーやPC）にインストールするソフトウェアです。OpenAEVからのインジェクト指示を受け取り、そのエンドポイント上で **実際に攻撃コマンドを実行** します。

```
┌──────────────┐          ┌──────────────┐
│  OpenAEV     │  指示    │ テスト対象     │
│  Platform    │────────→│ エンドポイント  │
│              │          │ ┌──────────┐ │
│              │  結果    │ │ Agent    │ │
│              │←────────│ └──────────┘ │
└──────────────┘          └──────────────┘
```

:::message alert
**重要：本番環境では絶対に使用しない**
エージェントは実際に攻撃コマンドを実行します。必ず **隔離されたテスト環境のVM** にのみインストールしてください。
:::

## 6.2 エージェントの種類

OpenAEVでは3種類のエージェントが利用可能です。

| エージェント | ライセンス | 対応OS | 用途 |
|------------|----------|--------|------|
| **OpenAEV Agent** | OSS (Apache 2.0) | Windows, Linux, macOS | 汎用。すべてのインジェクトに対応 |
| **Tanium** | 商用 | Tanium導入済み環境 | 大規模環境向け |
| **CrowdStrike** | 商用 | Falcon導入済み環境 | EDR自体の検知力テストにも |

:::message
**既存EDRをエージェントとして使う意味**
CrowdStrikeやTaniumをエージェントとして使う場合、「攻撃コマンドを実行 → 同じEDRが検知するか？」という **自己テスト** が可能になります。EDRベンダーが主張する検知率を、自社環境で実証できるわけです。
:::

## 6.3 エージェントのセットアップ

OpenAEV Agentの導入手順を示します。

### インストールと起動

```bash
# Linux の場合
chmod +x openaev-agent-linux-x86_64
./openaev-agent-linux-x86_64 \
  --url https://your-openaev-instance.example.com \
  --token <AGENT_TOKEN>

# Windows の場合 (PowerShell)
.\openaev-agent-windows-x86_64.exe `
  --url https://your-openaev-instance.example.com `
  --token <AGENT_TOKEN>
```

エージェントが正常に接続されると、「資産」メニューにエンドポイントが表示されます。

### テスト環境の推奨構成

```
┌─────────────────────────────────────────┐
│           テストネットワーク（隔離）        │
│                                         │
│  ┌───────────┐  ┌───────────┐          │
│  │ Windows   │  │ Linux     │          │
│  │ テストVM   │  │ テストVM   │          │
│  │ Agent ✓   │  │ Agent ✓   │          │
│  └───────────┘  └───────────┘          │
│                                         │
└──────────────┬──────────────────────────┘
               │ HTTPS
        ┌──────┴──────┐
        │  OpenAEV    │
        │  Platform   │
        └─────────────┘
```

:::message
テスト環境のVMは **スナップショットを取ってから** テストを実行し、テスト後に **スナップショットに復元** する運用を推奨します。
:::

## 6.4 アトミックテストとは

シナリオが「一連の攻撃フロー全体」を再現するのに対し、アトミックテストは **「1つの攻撃技法（テクニック）を単独でテスト」** するものです。

| | シナリオ | アトミックテスト |
|---|---------|------------|
| **粒度** | 攻撃フロー全体 | テクニック1つ |
| **目的** | 特定APTへの防御力を総合評価 | 特定テクニックの検知力確認 |
| **所要時間** | 数分〜数十分 | 数秒〜数分 |
| **使うタイミング** | 定期テスト、新脅威への対応 | ルール追加後の検証、日常チェック |

## 6.5 アトミックテストの実例

### AMSI Bypass — AMSI InitFailed

| 項目 | 値 |
|------|-----|
| **ATT&CK** | T1562.001 (Impair Defenses: Disable or Modify Tools) |
| **内容** | Windows AMSIを無効化する手法のテスト |

AMSI（Antimalware Scan Interface）はWindowsがマルウェア検知のために提供するAPIです。テスト結果が「インジェクト実行」なら、AMSIバイパスが成功しており、EDRのAMSI保護が不十分であることを意味します。

### その他の主要テスト

- **PowerShell コマンドライン攻撃** (T1059.001) — 不正なPowerShellコマンドの実行テスト
- **ローカルアカウントの作成** (T1136.001) — バックドアとしてのアカウント作成テスト
- **管理者一覧の列挙** (T1087.001) — 内部偵察フェーズのテスト
- **代替プロトコルによるデータ流出** (T1048) — DNSトンネリング等のデータ流出テスト

## 6.6 テスト結果の詳細分析

### グローバルスコア

アトミックテストの結果には **グローバルスコア** が表示されます。これは「防御がどの程度機能したか」を0〜100%で示すスコアです。

### テスト結果の対比

同じテストを複数のエンドポイントで実行すると、「どのマシンで防御が弱いか」が明確になります。

| テスト | oaev-win-01 | oaev-win-02 | oaev-linux-01 |
|-------|-------------|-------------|---------------|
| AMSI Bypass | ❌ バイパス成功 | ✅ ブロック | — (N/A) |
| PowerShell攻撃 | ❌ 実行成功 | ❌ 実行成功 | — (N/A) |
| アカウント作成 | ✅ 検知 | ✅ 検知 | ❌ 検知せず |
| データ流出 | ❌ 成功 | ✅ ブロック | ❌ 成功 |

この表から「oaev-win-01のEDRポリシーが弱い」「Linuxのアカウント作成監視が不十分」といった具体的な対策が見えてきます。

## 6.7 テスト結果を改善につなげる

```
テスト失敗（防御ギャップ発見）
│
├── 1. 原因の特定
│   ├── EDRルールが存在しない？ → ルール追加
│   ├── 閾値が高すぎる？ → 閾値調整
│   ├── ファイアウォールに穴がある？ → ポリシー修正
│   └── パッチ未適用？ → 即時パッチ
│
├── 2. 修正を実施
│
├── 3. 同じテストを再実行
│   └── 今度は「たぶん防げた」になることを確認
│
└── 4. 結果を記録（所見タブ / 学んだことタブ）
```

:::message
**アトミックテストはCI/CDのユニットテストと同じ**
アトミックテストは「セキュリティ版のユニットテスト」です。個々のルールが正しく機能するかをピンポイントで検証し、変更後のリグレッションを検出します。シナリオはE2Eテストに相当します。
:::

## 6.8 ペイロード管理

OpenAEVの「ペイロード」メニューでは、テストに使用する攻撃ペイロードを管理します。

| カテゴリ | 内容 |
|---------|------|
| **File Drop** | マルウェアファイルのダウンロード・実行 |
| **Command Execution** | OS コマンドの実行 |
| **Network** | 不正な通信の生成 |
| **Credential** | 資格情報の窃取テスト |
| **Cleanup** | テスト後のクリーンアップ |

ペイロードはXTMハブからインポートすることも、自作することも可能です。

## 6.9 OpenCTI × OpenAEV統合 — 双方向連携

OpenCTIとOpenAEVは **コネクタ** を通じて双方向にデータを同期します。

```
┌──────────────────┐                    ┌──────────────────┐
│    OpenCTI        │                    │    OpenAEV        │
│  脅威アクター ─────│──────────────────→│── シナリオ        │
│  TTP (ATT&CK) ───│──────────────────→│── インジェクト     │
│  CVE ────────────│──────────────────→│── Nucleiスキャン  │
│                    │                    │                    │
│  検証済みラベル ←──│──────────────────│── テスト結果      │
│  防御スコア ←─────│──────────────────│── スコアカード     │
└──────────────────┘                    └──────────────────┘
```

### OpenCTI → OpenAEV（脅威情報から攻撃シナリオへ）

1. OpenCTIでThreat Actor「Salt Typhoon」を検索
2. 関連するAttack Pattern（T1059, T1547, T1548, T1562）を確認
3. 関連するVulnerability（CVE-2024-20353, CVE-2021-26855, ...）を確認
4. OpenAEVに自動同期され、対応するシナリオ/インジェクトが利用可能になる

### OpenAEV → OpenCTI（検証結果のフィードバック）

テスト結果はOpenCTIにフィードバックされます。

- 脅威アクターに「テスト済み / 未テスト」のラベルが付く
- TTPごとに「自社で防御可能 / 要対策」のステータスが付く
- CVEに「パッチ適用済み / 未適用（確認済み）」の情報が追加される

これにより、OpenCTIの脅威情報に **「うちの環境では実際にどうなのか」** という文脈が追加されます。

## 6.10 コネクタの設定

```yaml
# docker-compose.yml への追加
connector-openaev:
  image: opencti/connector-openaev:latest
  restart: always
  environment:
    - OPENCTI_URL=http://opencti:8080
    - OPENCTI_TOKEN=${OPENCTI_ADMIN_TOKEN}
    - CONNECTOR_ID=${CONNECTOR_OPENAEV_ID}
    - CONNECTOR_NAME=OpenAEV
    - CONNECTOR_SCOPE=openaev
    - OPENAEV_URL=http://openaev:8080
    - OPENAEV_TOKEN=${OPENAEV_API_TOKEN}
    - CONNECTOR_LOG_LEVEL=info
  depends_on:
    - opencti
```

## 6.11 実践ワークフロー: 新しい脅威への対応

### Step 1-2: 脅威の確認と深掘り（OpenCTI）

新しい脅威レポートが入ったら、OpenCTIでThreat Actor、Vulnerability、Attack Pattern、Indicatorを確認します。

### Step 3: 防御をテストする（OpenAEV）

シナリオを実行し、CVEスキャンとTTP再現を実施します。

### Step 4-5: ギャップを特定して修復

| ギャップ | 修復アクション | 担当 | 緊急度 |
|---------|-------------|------|-------|
| CVE未パッチ | パッチ適用 | インフラチーム | **緊急** |
| T1059 検知不足 | SIEMにPowerShell検知ルール追加 | SOCチーム | 高 |
| T1562 検知不足 | EDRポリシーのDefense Evasion強化 | セキュリティエンジニア | 高 |

### Step 6: 再テストして確認（OpenAEV）

修復完了後、同じシナリオを再実行して全ステップが防御/検知されることを確認します。

:::message
**このサイクルの所要時間**
手動で全ステップを実施すると数日〜数週間かかりますが、OpenCTI + OpenAEVの統合環境では **数時間〜1日** で完了できます。
:::

## 6.12 定期テストの運用

| テスト種類 | 頻度 | 目的 |
|----------|------|------|
| アトミックテスト（全テクニック） | **週次** | EDR/SIEMルールのリグレッション検出 |
| 主要シナリオ（Salt Typhoon等） | **月次** | APT対応力の定点観測 |
| 全シナリオ一括 | **四半期** | 総合的な防御力評価 |
| 新脅威シナリオ | **随時** | 新しいAPT/CVEへの即座の対応 |

## 6.13 ATT&CKカバレッジマトリクス

OpenAEVのテスト結果をMITRE ATT&CKマトリクスにマッピングすると、防御の「穴」が視覚的に把握できます。

```
MITRE ATT&CK マトリクス（防御カバレッジ）

          Initial   Execution  Persistence  Priv.Esc  Def.Evasion
          Access
T1190     ❌        -          -            -         -
T1059     -         ❌         -            -         -
T1547     -         -          ✅           -         -
T1548     -         -          -            ✅        -
T1562     -         -          -            -         ❌

✅ = 防御/検知可能（テスト済み）
❌ = ギャップあり（要対策）
-  = 未テスト
```

このマトリクスを定期的に更新することで、組織の防御カバレッジの向上を可視化できます。経営層への報告にも有効です。

## 6.14 統合の価値まとめ

### 統合前（分断された運用）

CTIチーム → レポートを読む → メールで共有 → 数週間〜数ヶ月後にペンテストで確認（年1回）

### 統合後（Threat-Informed Defense）

OpenCTI → 自動で脅威情報を収集 → OpenAEV → 自動でテスト → 結果をフィードバック → ギャップを即座に特定 → 修復 → 再テスト → 確認

:::message
**Threat-Informed Defenseの本質**
「脅威を知る」→「防御を検証する」→「ギャップを埋める」→「再検証する」— このサイクルを継続的に回すことが、Threat-Informed Defenseの本質です。OpenCTIとOpenAEVはこのサイクルを1つのエコシステム上で完結させます。
:::

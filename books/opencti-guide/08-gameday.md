AWSの障害訓練「Game Day」、Netflixの「Chaos Engineering」— 意図的に障害を起こして対応力を鍛える文化はインフラの世界では定着しています。セキュリティでも同じことができます。OpenCTIとOpenAEVを使えば、 **実在のAPTグループの攻撃を再現し、SOCチームの対応力をゲームデイ形式で鍛える** ことが可能です。

この章では、ゲームデイの企画から実施、振り返りまでを実践的に解説します。

## 8.1 セキュリティ・ゲームデイとは

セキュリティ・ゲームデイは、実際の攻撃シナリオを **制御された環境で再現** し、SOCチームやインシデント対応チームの検知・対応能力を検証するイベントです。

### 従来のセキュリティ訓練との違い

| | 従来の訓練 | ゲームデイ（OpenAEV） |
|---|---------|------------------|
| **攻撃の再現** | 机上の想定、または外部Red Teamに委託 | OpenAEVが自動で実行 |
| **コスト** | Red Team委託で数百万円〜 | OSS版は無料 |
| **頻度** | 年1-2回 | 月次〜四半期で実施可能 |
| **再現性** | Red Teamのスキルに依存 | 同じシナリオを何度でも再現 |
| **計測** | 定性的な報告書 | 4軸スコアで定量評価 |
| **準備期間** | 数週間〜数ヶ月 | 数時間〜数日 |

### 3つのゲームデイ形式

**形式1: テクニカル・ゲームデイ（Purple Team演習）** — OpenAEVのエージェントを使ってテスト環境に実際の攻撃コマンドを投げ、SOCチームがリアルタイムで検知・対応する形式です。

**形式2: 卓上演習（Tabletop Exercise）** — 技術的な攻撃は実行せず、人的インジェクトを使ってシナリオを進行します。組織レベルの意思決定を訓練します。

**形式3: ハイブリッド（Full-Scale Exercise）** — テクニカル + 卓上を組み合わせた最も本番に近い形式です。

## 8.2 ゲームデイの企画

### Step 1: 脅威の選定（OpenCTIで調査）

| 基準 | 内容 | 例 |
|------|------|-----|
| **業界の関連性** | 自社の業界を標的とするAPTか | 通信業界 → Salt Typhoon |
| **最近の活動** | 直近で活発に攻撃しているか | OpenCTIのReportsで確認 |
| **技法の網羅性** | 多様なKill Chain Phaseをカバーするか | 初期侵入〜データ流出まで |
| **テスト環境の対応** | 自社のテスト環境で再現可能か | Windows/Linux環境の有無 |

### Step 2: スコープと目標の定義

| 項目 | 定義すること |
|------|-----------|
| **形式** | テクニカル / 卓上 / ハイブリッド |
| **参加者** | SOCアナリスト、セキュリティエンジニア、マネジメント |
| **対象環境** | テスト環境のVM群（本番環境は絶対に対象外） |
| **成功基準** | MTTD、MTTRの目標値 |
| **所要時間** | 2時間〜半日が一般的 |

:::message alert
**鉄則：本番環境では絶対にやらない**
ゲームデイの攻撃シミュレーションは、必ず **隔離されたテスト環境** で実施してください。
:::

### Step 3: 参加者への通知

| 通知レベル | 対象者 | 伝える内容 |
|----------|--------|----------|
| **フル通知** | ゲームデイ運営チーム | 全詳細 |
| **概要のみ** | マネジメント層 | 日程、目的、影響範囲 |
| **事前通知なし** | SOCアナリスト | 知らせない（本番と同じ条件） |

:::message
SOCアナリストに事前通知しないのが最も効果的です。初回は事前通知ありで実施し、慣れてきたら通知なしに移行するのも良いアプローチです。
:::

## 8.3 テクニカル・ゲームデイの実施

### タイムラインの例（Salt Typhoonシナリオ、3時間）

```
09:00  [運営] テスト環境の最終確認、エージェント起動
09:15  [運営] OpenAEVでSalt Typhoonシナリオを起動
       [OpenAEV] Phase 1 開始: Nuclei CVEスキャン（6件）
09:20  [OpenAEV] Phase 2 開始: エージェントによるTTP再現
       [SOC] ← アラートが飛び始める
09:25  [SOC] EDRアラート「不審なファイルダウンロード」を検知
       → トリアージ開始
09:40  [SOC] SIEMアラート「PowerShell encoded command」を検知
       → 関連IOCをOpenCTIで照合
10:00  [SOC] インシデントとして認定 → マネジメントへの第一報
10:30  [SOC] 影響範囲の特定完了 → 封じ込め策を実施
11:00  [SOC] 根本原因の特定 → 復旧手順の策定
11:30  [運営] ゲームデイ終了宣言
12:00  [全員] 振り返り（Hot Wash）
```

## 8.4 卓上演習の実施

### 卓上演習のシナリオ例（Akira Ransomware Crisis）

```
13:00  [ファシリテーター] シナリオ説明
       「金曜17時、経理部門のファイルサーバーが暗号化された。
        Akiraランサムウェアと思われる。身代金要求あり。」

13:10  [インジェクト #1] → 参加者に状況通知メール送信
       質問: 「最初に何をしますか？」

13:20  [議論] 初動対応
       ├── ネットワーク隔離するか？
       ├── バックアップの状態は？
       └── 法執行機関に連絡するタイミングは？

13:35  [インジェクト #2] 「他の部門のファイルサーバーも暗号化開始」

13:45  [議論] エスカレーション判断
       ├── CEOへの報告タイミング
       ├── 広報への連絡
       └── 身代金を払うか？

14:00  [インジェクト #3] 「記者から問い合わせ」

14:15  [議論] 広報対応

14:30  [振り返り] 学んだことの共有
```

```
┌─────────────────────────────────────────────────┐
│              ハイブリッド演習                       │

│  [テクニカル層]              [マネジメント層]       │
│  ┌──────────────┐          ┌──────────────┐      │
│  │ OpenAEV      │          │ 卓上演習      │      │

│  │ 検知→対応    │←─────────│ 広報・法務    │      │
│  └──────────────┘  指示     └──────────────┘      │

```

## 8.6 振り返りと改善

### 定量メトリクス

| メトリクス | 定義 | 目標値の例 |
|----------|------|----------|
| **MTTD** | 攻撃開始〜最初のアラート検知 | < 5分 |
| **MTTA** | アラート検知〜アナリストが着手 | < 10分 |
| **MTTI** | 着手〜影響範囲の特定完了 | < 30分 |
| **MTTR** | 着手〜封じ込め完了 | < 1時間 |
| **検知率** | インジェクト総数のうち検知できた割合 | > 80% |

### OpenAEVへの記録

- **テスト** タブ: 実行結果の自動記録
- **学んだこと** タブ: 振り返りのメモを手動記入
- **所見** タブ: 発見されたギャップと対策を記録

## 8.7 ゲームデイの成熟度モデル

```
Level 1: 卓上演習（事前通知あり）
│  → OpenAEVの人的インジェクトで進行
▼
Level 2: テクニカル（事前通知あり）
│  → SOCに「今日ゲームデイやります」と通知
▼
Level 3: テクニカル（事前通知なし）
│  → 本番と同じ緊張感で対応力を測定
▼
Level 4: ハイブリッド（事前通知なし）
│  → 組織全体の対応力を総合評価
▼
Level 5: 継続的バリデーション
   → OpenAEVのスケジュール実行で毎週自動テスト
   → 異常があればSlack通知 → SOCが対応
```

:::message
**Level 5 が最終目標**
ゲームデイを「年に数回のイベント」から「毎週自動で回るプロセス」に進化させるのが理想です。
:::

## 8.8 コンテナ環境でのゲームデイ

Kubernetes環境でのゲームデイも有効です。

### シナリオ例：コンテナへの攻撃シミュレーション

```
[攻撃シナリオ: コンテナ侵入と横展開]
│
├── 1. 脆弱なコンテナイメージのデプロイ
│   └── 既知のCVEを含むイメージをテスト用Podに配置
│
├── 2. コンテナ内でのコマンド実行
│   └── Falcoルール: "Terminal shell in container" が発火するか？
│
├── 3. ServiceAccount Tokenの窃取
│   └── Falcoルール: "Read sensitive file" が発火するか？
│
├── 4. 横展開の試行
│   └── Sysdig Secure: "Unexpected K8s API call" が検知するか？
│
└── 5. データ流出の試行
    └── NetworkPolicyやSysdigが遮断するか？
```

### OpenCTIとの連携ポイント

1. **OpenCTI** でコンテナを標的とする脅威アクターのTTPを調査（例: TeamTNT、Kinsing）
2. そのTTPをKubernetes環境用のインジェクトに変換
3. OpenAEVまたはカスタムスクリプトで実行
4. Falco/Sysdig Secureのアラートを評価
5. 結果をOpenCTIにフィードバック

## 8.9 ゲームデイのチェックリスト

### 事前準備（1-2週間前）

- [ ] 脅威アクターの選定（OpenCTIで調査）
- [ ] シナリオの選択またはカスタム作成（OpenAEV）
- [ ] テスト環境の準備とスナップショット取得
- [ ] エージェントのインストールと動作確認
- [ ] SIEM/EDRの正常動作確認
- [ ] 参加者への通知（通知レベルに応じて）
- [ ] 成功基準とメトリクスの定義

### 当日

- [ ] テスト環境の最終確認
- [ ] OpenAEVでシナリオを起動
- [ ] 運営チームによるリアルタイム監視
- [ ] タイムスタンプの記録（MTTD/MTTR計測用）
- [ ] 終了宣言 → Hot Wash（即時振り返り）

### 事後（1週間以内）

- [ ] OpenAEVスコアの集計
- [ ] メトリクスの算出
- [ ] 発見されたギャップのリスト化
- [ ] 改善アクションの策定と担当割り当て
- [ ] テスト環境のスナップショット復元
- [ ] 次回ゲームデイの日程設定

:::message
**まずは小さく始める**
いきなりハイブリッドのFull-Scale Exerciseを目指す必要はありません。まずはLevel 1の卓上演習、またはアトミックテスト数個を使った小規模テクニカル演習から始めて、チームの経験値を積んでいきましょう。
:::

### 次のステップ

1. **デモ環境で手を動かす** — OpenCTI（demo.opencti.io）とOpenAEV（demo.openaev.io）にFiligranアカウントでログインし、Salt TyphoonやLazarus Groupを検索して関連エンティティを辿ってみてください。

2. **小さなゲームデイを企画する** — アトミックテスト3-5個で構成した小規模テクニカル演習を、テスト用VMで実施してみてください。

3. **APIを試す** — pyctiでIOCを取得するスクリプトを書き、自社のSIEMに連携するPoC（概念実証）を作ってみてください。

4. **自社環境を構築する** — Docker ComposeでOpenCTI + OpenAEVの環境を立ち上げ、コネクタの追加やエージェントのテストを試してみてください。

5. **Threat-Informed Defenseサイクルを回す** — 「知る → 検証する → 修復する → 再検証する」を自社の運用に組み込んでみてください。

:::message
導入当初は「完璧さ」を目指さず、「継続性」を重視してください。週1回の定期会議で運用課題を共有し、段階的に改善することが成功の鍵です。
:::
